# Demo Doc Generator

グロースパック for LINEのデモ構築用ドキュメントを自動生成するスキル。

## ワークフロー

```
1. 情報収集 → 2. フレームワーク適用 → 3. ドキュメント生成 → 4. レビュー
```

### Step 1: 情報収集

以下のソースから情報を収集:

1. **LPサイト** - WebFetchでLPのURLを読み込む
2. **プレスリリース** - 公式発表から価値提案を抽出
3. **既存デモ** - 参考デモがあれば確認

```
# 例: LPサイトの読み込み
WebFetch: https://lp-coupon-function.vercel.app
→ コンセプト、機能一覧、ターゲット顧客を抽出
```

### Step 2: フレームワーク適用

[references/frameworks.md](references/frameworks.md) を参照し、以下のフレームワークを適用:

| フレームワーク | 目的 |
|---------------|------|
| JTBD | ユーザーの文脈と動機を言語化 |
| VPC | 痛みと解決策のマッチング |
| リーンキャンバス | ビジネスモデル全体の整理 |
| ユーザーストーリー | MVP機能の定義 |
| 狩野モデル | 機能の優先順位付け |

### Step 3: ドキュメント生成

`assets/templates/` のテンプレートを使用して以下を生成:

```
docs/
├── 01-jtbd.md           # Jobs To Be Done
├── 02-vpc.md            # バリュー・プロポジション・キャンバス
├── 03-lean-canvas.md    # リーンキャンバス
├── 04-user-stories.md   # ユーザーストーリーマッピング
├── 05-kano-model.md     # 狩野モデル
└── 06-prototype-guide.md # プロトタイプガイドライン
```

**出力先**: リポジトリのルートに `docs/` ディレクトリを作成（サブディレクトリではなく、必ずルート直下に配置）

**06-prototype-guide.md** には以下を含める:
- 画面一覧と状態シミュレーション要件
- demoSteps設定（demo-builderで直接使用可能な形式）
- デモシナリオ表
- アニメーション要件
- 魅力的品質機能の詳細（ビジュアル処理含む）

### Step 4: レビュー

生成後、以下を確認:
- MVPスコープが明確か
- 魅力的品質が1つに絞られているか
- プロトタイプガイドのブランドカラーが設定されているか

## クイックスタート

```
ユーザー: 「このLPからデモドキュメントを作成して」
         URL: https://lp-xxx.vercel.app

1. WebFetchでLP読み込み
2. 読み込んだ情報を元にJTBD作成
3. VPC作成
4. リーンキャンバス作成
5. ユーザーストーリー作成
6. 狩野モデルでMVP機能決定
7. プロトタイプガイド作成
```
