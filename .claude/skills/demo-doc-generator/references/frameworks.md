# プロダクト設計フレームワーク

## JTBD (Jobs To Be Done)

ユーザーがアプリを使いたくなる「文脈」と「動機」を言語化する。

### 構成要素

| 要素 | 説明 | 例 |
|------|------|-----|
| When | どんな状況で | 店舗で買い物をしている時 |
| I want to | 何をしたいか | お得なクーポンを簡単に使いたい |
| So that | どんな成果を得たいか | 会計時に割引を受けられる |

### 出力形式

```
When [状況]
I want to [行動]
So that [成果]
```

## バリュー・プロポジション・キャンバス (VPC)

ユーザーの「痛み」と解決策をマッチングさせる。

### 顧客プロファイル

| 要素 | 説明 |
|------|------|
| Customer Jobs | 顧客が達成したいタスク |
| Pains | 達成を妨げる障害・不満 |
| Gains | 達成時に期待する成果 |

### バリューマップ

| 要素 | 説明 |
|------|------|
| Products & Services | 提供する製品・サービス |
| Pain Relievers | 痛みをどう解消するか |
| Gain Creators | 成果をどう実現するか |

## リーンキャンバス

ビジネスモデル全体を1枚で整理する。

### 9つの構成要素

1. **課題 (Problem)**: 解決する上位3つの課題
2. **顧客セグメント (Customer Segments)**: ターゲット顧客
3. **独自の価値提案 (Unique Value Proposition)**: 競合との差別化ポイント
4. **ソリューション (Solution)**: 課題への解決策
5. **チャネル (Channels)**: 顧客へのリーチ方法
6. **収益の流れ (Revenue Streams)**: 収益モデル
7. **コスト構造 (Cost Structure)**: 主要なコスト
8. **主要指標 (Key Metrics)**: 成功を測る指標
9. **圧倒的優位性 (Unfair Advantage)**: 模倣困難な強み

## ユーザーストーリーマッピング

ユーザー体験を時系列で整理し、MVP機能を定義する。

### 構造

```
[ユーザーアクティビティ]
    ↓
[ユーザータスク]
    ↓
[ユーザーストーリー] → MVPライン
```

### ユーザーストーリー形式

```
As a [ユーザー種別]
I want to [機能・行動]
So that [得られる価値]
```

## 狩野モデル

機能の優先順位付けに使用。MVPでは「当たり前品質」の確保と1つの「魅力的品質」に集中。

### 品質分類

| 品質タイプ | 説明 | MVP判断 |
|-----------|------|---------|
| **当たり前品質** (Must-be) | ないと不満、あっても満足度上がらない | 必須 |
| **一元的品質** (Performance) | あればあるほど満足度が上がる | 選択的 |
| **魅力的品質** (Delighter) | なくても不満はないが、あると感動 | 1つに集中 |

### 例

- 当たり前: ログイン機能、エラーがないこと
- 一元的: 処理速度、レスポンス時間
- 魅力的: AIによる予期せぬ便利な提案
