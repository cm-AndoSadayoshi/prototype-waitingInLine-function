# 狩野モデル（機能優先順位）

## プロダクト名: LINE順番待ち

---

## 品質分類

### 当たり前品質 (Must-be) - 必須

ないと不満だが、あっても満足度は上がらない機能。

| 機能 | 説明 | MVP判断 |
|------|------|---------|
| QRコード整理券発行 | 店頭QRスキャンで整理券を取得できる | ✅ 必須 |
| 整理券番号表示 | 自分の整理券番号が確認できる | ✅ 必須 |
| 現在の待ち組数表示 | あと何組待っているか確認できる | ✅ 必須 |
| LINE通知（呼び出し） | 順番が来たら通知が届く | ✅ 必須 |
| エラーのない動作 | アプリがクラッシュしない、正確に動作する | ✅ 必須 |

---

### 一元的品質 (Performance) - 選択的

あればあるほど満足度が上がる機能。

| 機能 | 説明 | MVP判断 |
|------|------|---------|
| リアルタイム更新 | 待ち組数がリアルタイムで更新される | ⚪ MVP採用 |
| 順番接近通知 | 3組前など事前に通知が届く | ⚪ MVP採用 |
| 待ち時間予測 | 予想待ち時間を表示する | ⚪ 検討 |
| 複数店舗対応 | 複数店舗の順番待ちを管理できる | ⚪ 後回し |
| プッシュ通知の確実性 | 通知が確実に届く（再通知機能など） | ⚪ 検討 |

---

### 魅力的品質 (Delighter) - 1つに集中

なくても不満はないが、あると感動する機能。

| 機能 | 説明 | MVP判断 |
|------|------|---------|
| **待機中の会員登録導線** | 待ち時間を活用して自然にLINE会員登録ができる | ⭐ MVP採用 |
| 待機中のクーポン配布 | 待機お礼としてクーポンがもらえる | 後回し |
| 来店ポイント付与 | 順番待ち利用でポイントが貯まる | 後回し |
| AIによる待ち時間提案 | 混雑予測に基づく来店時間の提案 | 後回し |
| 席種・人数の事前指定 | 希望席種を事前に指定できる | 後回し |

---

## MVP機能サマリー

### 採用機能

| 品質タイプ | 機能 |
|------------|------|
| 当たり前品質 | QRコード整理券発行、整理券番号表示、待ち組数表示、LINE呼び出し通知 |
| 一元的品質 | リアルタイム更新、順番接近通知 |
| 魅力的品質（1つ） | **待機中の会員登録導線** |

### 見送り機能

| 機能 | 理由 |
|------|------|
| 待ち時間予測 | 精度確保に複雑なロジックが必要、MVP後に検討 |
| 複数店舗対応 | 単一店舗での価値実証を優先 |
| 待機中クーポン配布 | 会員登録導線を魅力的品質として優先、追加機能として後日検討 |
| 席種・人数指定 | カスタマイズ要素が多く、MVP後の拡張機能として提供 |
| 来店ポイント付与 | グロースパック他アセットとの連携が必要、段階的に実装 |

---

## MVP設計の方針

### 魅力的品質の選定理由: 「待機中の会員登録導線」

1. **グロースパックの価値提案との整合性**: 「待ち時間を会員化のチャンスに変える」がプロダクトの核心
2. **差別化ポイント**: 単なる順番待ちシステムではなく、CRMツールとしての価値
3. **実装コストと効果のバランス**: 比較的シンプルな実装で高い効果が期待できる
4. **他アセットとの連携**: 会員証・セグメント配信など既存アセットとの相乗効果

### 当たり前品質の徹底

- 整理券発行〜呼び出し通知の基本フローは**完璧に動作**することが前提
- エラーや不具合があると顧客体験が大きく損なわれる
- MVP段階でも品質に妥協しない
